## v5 — 2026-07-07
- **New profile: `template-solo/` (34 files) — single-session vibe (user request).** The main session = the orchestrator; product code is neither written nor read in the main context — three subagents carry it: **scout** (read-only code Q&A, compact findings + file:line) · **implementer** (one part step: code + tests + checkpoint commit, compact report) · **verifier** (GATE 4, same file as orchestrated). New commands: **/part** (drives a part through spec→gates→impl→verify→merge, stopping at every 🚦) · **/tidy** (the context-hygiene ritual: sync docs to ground truth + rotation + sweep chat-only decisions → declare the reset options). **PreCompact is default-ON in solo.** No phase-kickoffs.md and no /new-part — the kickoff/handover concepts are meaningless in solo; instead the **restart test**: after `/clear`, resume from CLAUDE.md + docs + memory (memory survives `/clear`). The honest mechanism: the model cannot wipe its own context — prevention (delegation-first, rule 12) + safe reset (/tidy, rule 13; the human runs /compact–/clear) + the safety net (PreCompact). The §4.3 boundary is deliberately inverted in solo: development goes to subagents, the user watches compact reports; ground-truth verification lives on as the post-dispatch git check.
- **New §17:** the two-profile comparison table + solo architecture + the three context-hygiene layers + **profile switching** (living-docs are identical in both → mid-project solo↔orchestrated switching is supported; start solo, switch when parallelism appears). §14 marked as the "orchestrated profile"; the header gained the v5 profile line.
- **Shared-DNA byte parity:** 4 hooks + verifier + the spec/plan/checkpoint/gate3 commands + doc skeletons are exact copies from `template/` (verified by diff — two profiles derived from one source, no fork drift).
- **README.md (English):** the two-profile comparison + setup and daily usage + hook/command reference + versioning; the repo's front door. `template/` version markers bumped to v5 (with the orchestrated-profile label).
- **Solo autonomy (user request):** (a) **wide permissions** — Edit/Write + `git commit/merge` in allow (push/reset/rebase ask, destructive ops deny); the human is never stopped between gates — the control points are the GATES, not permission prompts; the invariants live in hooks; (b) **batched dispatches** — a small part = the whole approved plan in one implementer dispatch, bigger = 2–4-step packages; a checkpoint commit per step (granular history, few interruptions); (c) **the `/clear` recommendation at part boundaries** — the solo equivalent of orchestrated's fresh session; /tidy now declares the reset options. The honest limit spelled out: the model cannot reset its own context (/clear–/compact are user commands); what it can do is prevention + making the reset zero-risk + a one-keystroke offer; quality order: boundary-/clear > /compact > auto-compact.
- **guard-env hardened + verified (user request, both templates):** closed bypasses — globs (`cat .env*`, `.env.*`), no-space redirects (`>.env`), case variants (`cat .ENV` — the macOS FS is case-insensitive), the Grep tool's `glob` parameter, subdirectory paths (`api/.env`); the `source/cp/mv/base64/python-open/variable-assignment` paths were already caught, now proven by tests. Only `.env.example/sample/template` are fully accessible. **The block-behavior rule (CLAUDE.md 3):** on a guard-env block STOP — NO retries via another path, report to the user + wait for approval; env values are always placed by the human. The settings deny list expanded to `**/.env*` variants (6 names, subdirectories included). **A 28-scenario suite passes 28/28 on both templates** (18 bypasses blocked, 10 legitimate uses stay free — process.env / the dotenv package / .environment.ts with zero false positives); the older 18-test suite passes unchanged.

## v4 — 2026-07-04
> Theme: "optimum > perfect" — mechanisms that reduce the human load were added; ones that add ceremony deliberately were not.
- **§9 leak protocol + the secret-scan hook (CLAUDE.md rule 3 extended):** if the user pastes a sensitive value into chat (DB URL, API key…), the model does not repeat it / does not write it into any file-doc-command; if it landed anywhere it deletes it immediately; it notifies + **recommends rotation** (the honest limit: chat history cannot be truly deleted — the only permanent fix is rotation). New **UserPromptSubmit hook `secret-scan.sh`**: detects common shapes (userinfo conn-strings · sk-/AKIA/ghp_/github_pat_ · PRIVATE KEY · JWT · key=value) and reminds the protocol at exactly that moment — without blocking (false-positive cost ~zero).
- **§9 context discipline (new CLAUDE.md rule 9):** a mid-phase user question gets the SHORT answer requested — no re-dumping the whole phase flow with every question; the full summary only when asked or at a gate.
- **§3 + §9 working-tree isolation (new CLAUDE.md rule 10):** parallel sessions (client/backend etc.) cannot share one working tree — the root of conflicts/code loss. One active session per directory; parallel parts in **separate git worktrees**; the Manager partitions parts with **disjoint file scopes** (overlapping ones run sequentially); a parallelism check added to `/new-part`; checkpoint commit before branch switches.
- **§9 the language rule (new CLAUDE.md rule 11) + templates fully English:** chat follows the user (Turkish fine); ALL documentation (living-docs, specs, runbooks, commit messages, code comments) in ENGLISH — rationale: fewer tokens · stronger instruction-following · consistent terminology across sessions. The template's 32 files (doc skeletons, command/agent prompts, hook comments+messages) translated; the mechanics (file names, regexes, the `.gate4-ok` path, markers) unchanged; hook tests re-run (18/18 + secret-scan 3/3). Terminology: KAPI→GATE, parça→part, devir→handover; the design numbering G-N kept (GATE is never abbreviated to a single letter — no collision).
- **④ Design-track tooling:** the design decision-maker is **Claude Design** (connects to Claude Code via MCP); the connection requires Claude Code **from the terminal (CLI)** — recorded in §1/§13 + the template (workflow, STARTGUIDE, design STATUS).

## v3 — 2026-07-04
- **`template/` materialized (the biggest change):** the §14 skeleton is no longer prose but real files — a LEAN CLAUDE.md (8 critical rules) + workflow.md (**the normative summary**: rules only, rationale in the playbook — the kernel/rationale split emerged without restructuring the playbook) + STARTGUIDE (setup + a ready Phase 0 kickoff) + living-doc starters + the spec template (with a gate-profile field) + a runbook template + `.claude/` (settings + 3 hooks + the verifier agent + 6 commands) + `memory-seed/`. The bootstrap-drift source is closed: no more re-generation from prose — copy & use.
- **§9 main-guard + /gate3 (applying "hook > instruction"):** "nothing enters main without GATE 4" is now physical — code commits on main and unmarked merges are blocked (docs-only commits stay free; the marker = `.claude/.gate4-ok` after human approval). The `/gate3` command produces mechanical evidence for GATE 3 (test/lint/typecheck). The principle: the right hook removes a human vigil, it does not add ceremony. *(Both hooks passed an 18-scenario test suite.)*
- **§2 per-part gate profile:** for a small/low-risk part GATES 1+2 combine into one approval (a proactive answer to gate fatigue; the §15 retro calibration remains as the reactive layer); never on money/auth/data-loss surfaces. The profile is written into a spec field.
- **New §2.1 cancel/rollback:** part cancellation (branch abandoned + a one-line record) · plan falls apart mid-IMPL → delta-plan back to GATE 2 · GATE 4 finds a spec flaw → back to GATE 1.
- **§9 anti-confabulation generalized:** not only Ops — every session that has been through compaction says "not in the docs → not recorded".
- **§10 the handover test (the Manager symmetric of the Ops test):** information in the handover prompt that is missing from the docs = a docs gap; the doc is fixed, not the prompt.
- **§11 memory mechanism note:** memory is project-directory-scoped and role-blind; role selectivity comes from the kickoff. The seed file moved to `memory-seed/` (the first Manager session saves it).
- **④ Design decision-maker = Claude Design** (MCP; terminal mandatory) recorded in §1/§13 + template.
- *Deliberately not added:* a two-document restructuring of the playbook (maintenance burden; the template already is the normative form) · an S/M/L part taxonomy (judgment suffices — the follower is a developer) · a stateless Manager (human friction; the long-lived Manager stays as cache, freshness is tested at handover) · extra ceremony hooks.
